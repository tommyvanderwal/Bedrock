## Regressions caught + fixed during this run

1. **Create/Convert/Delete validation was async** — invalid input
   (bad name, missing ISO, oversized vcpus/ram, duplicate name) got
   202 + task_id and then failed in the background task. UX smell:
   the caller has to poll /api/tasks to learn their input was wrong.
   Moved all sync-discoverable validation above the task-create
   call, so bad input now gets a proper `4xx` with a helpful
   detail string. E3 / E9 / E10 verify the fix.

2. **VM name regex requires ≥ 3 chars** — earlier noted but not
   mentioned: `^[a-z][a-z0-9-]{1,30}[a-z0-9]$` means min len 3 with
   letter start and alphanumeric end. Test case accidentally used
   2-char names and got consistent 400s, which proves the check
   works but the error message should probably mention "at least 3
   characters" explicitly. Noted as a minor UX improvement for a
   future pass — not blocking.

## Things that work reliably now

- Full rebuild: 4-node HA cluster from zero in **< 5 min** (most of
  which is WAN-bound dnf install, not Bedrock).
- Cattle ↔ pet ↔ ViPet, single- **and** multi-disk, with atomic
  rollback on per-disk failure.
- Live migrate across any pet/ViPet VM regardless of disk count
  (virsh migrate handles them all in one call; our code cycles
  `allow-two-primaries` + `drbdadm primary` per resource).
- Import: single disk (qcow2, raw, img, vhd, vhdx) and multi-disk
  OVA (tar + OVF parse + per-disk qemu-img; virt-v2v for Windows
  driver-injection path). Imported VMs are indistinguishable from
  fresh-create VMs after import — all actions (convert, migrate,
  settings, delete) apply cleanly.
- Task drawer shows per-step progress for every long action:
  create, delete, convert, create-from-import, import-convert.
  Finished tasks age out after 15 min so the drawer stays clean.
- The DRBD silent-truncation guard has never tripped in this run
  (good — it means the meta-LV formula is correct at every size
  tested). If it ever trips in the future, the operator will see
  exact bytes + a pointer at the meta_mb formula.

## Follow-ups noted but not fixed this run

- **Node-id collision on parallel joins**: three nodes joining in
  parallel each see `len(existing)=1` in cluster.json at the moment
  they query (before anyone else has written yet), so all three get
  `node_id=1`. Cosmetic only — unique node *names* (FQDNs) are the
  primary key everywhere. Worth fixing: serialize register writes
  with a lock or let mgmt assign the id.
- **VM-name regex message**: says "3-32 chars" via the `{1,30}` in
  regex but error message could be clearer about the required shape.
- **Multi-disk attach post-convert**: `POST /api/vms/{name}/disks`
  creates a new plain LV and attaches to the running VM, but for
  pet/ViPet VMs the new disk is cattle-local (not replicated). A
  follow-up convert of the newly-attached disk to DRBD would be a
  logical extension.
